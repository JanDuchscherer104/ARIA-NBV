1.1 ASE: Single Trajectory Per Scene

  • Each ASE scene provides only one pre-recorded egocentric trajectory. We cannot query arbitrary novel viewpoints.

    • EFM3D introduces GT meshes for 100 validation scenes. We can use these to compute oracle RRI for candidate views. Use the mesh predicted by EFM3D as pseudo-GT for other scenes.

  • depth maps in ASE dataset are ground-truth!

  • non-gt point clouds are sparse! How to sample a PC from the candidate view that can be compared to the pre-existing semi-dense SLAM PC that is provided in the ASE dataset?

  • Should NBV prediction output discrete selection (VIN-NBV) or continuous pose (GenNBV)?

    • Start with randomly sampled candidate views for which the RRI is computed. Action space is then the selection among these candidates.
    • Future work: learn to generate continuous poses directly, whose fitness can than be quantified using the RRI predictor; here we can also consider the free space voxels that are used in EFM3D to avoid collisions.

# Issues & Questions

## ASE: Single Trajectory Per Scene

- Each ASE scene provides only one pre-recorded egocentric trajectory. We cannot query arbitrary novel viewpoints.
    - EFM3D introduces GT meshes for 100 validation scenes. We can use these to compute oracle RRI for candidate views. Use the mesh predicted by EFM3D as pseudo-GT for other scenes.

- depth maps in ASE dataset are ground-truth!

- non-gt point clouds are sparse! How to sample a PC from the candidate view that can be compared to the pre-existing semi-dense SLAM PC that is provided in the ASE dataset?
- Should NBV prediction output discrete selection (VIN-NBV) or continuous pose (GenNBV)?
  - Start with randomly sampled candidate views for which the RRI is computed. Action space is then the selection among these candidates.
  - Future work: learn to generate continuous poses directly, whose fitness can than be quantified using the RRI predictor; here we can also consider the free space voxels that are used in EFM3D to avoid collisions.


# Action Items


## HIGHEST PRIO:

- implement the `OracleRRI` class with PC to mesh distance computation between two point clouds
  - Mesh to distance field computation??
  - How can we consider only relevant (observable parts) of the scene, such that we don't have to consider the entire mesh (e.g. there might be different rooms that cannot be considered)
  - must be implemented as `TorchMetric`
- implement `CandidateViewGenerator`, that generated N candidate SE(3) poses (roll = 0) around the latest camera position in the trajectory. Must generate poses of type `PoseTW` (`external/efm3d/efm3d/aria/pose.py`)
- implement a `CandidateSample` that takes the candidate poses of type `PoseTW` and samples rays from the corresponding view onto the GT mesh to obtain a dense point cloud from this candidate view.
  - candidate pose generation: only within free space and within confines of the room (based on SSL for training); based on predicted occupancy from EFM3D for eval
  - ray mesh intersection: check if both poses can see each other
  - sampling based on occupancy grid?

- implement a class (name to be decided) that fuses the semi-dense SLAM PC $\mathcal{P}_t$ with the dense PC from the candidate view  $\mathcal{P}_{\mathbf{q}}$ to obtain the updated PC $\mathcal{P}_{t \cup \mathbf{q}}$.
- crop GT mesh to relevant volume around the trajectory to speed up computation, this is especially relevant for completeness computation.


### Data Handling

- Implement custom data handling for the ASE dataset, using the tools provided in ATEK:
  - Should provide easy access to the ASE dataset in ATEK WDS format together with the ground truth meshes.
  - Should wrap around their existing dataset and dataset downloader implementations to provide easy acces to the data we need.
  - Download all 3 forms of the dataset for scenes for which the GT meshes are available.
  - Figure out, how to best manage to download corresponding data files (ATEK, raw ASE, GT meshes) for the same scenes.
  - We also want to represents scenes together with the snippets that reside in them (snippets are in atek dataset structure ready for training).
  - metadata should be comined in a single file for easy access.
  - should provide mapping from scene_id to snippet_ids and vice versa.
  - should leverage their download scripts internally to provide a cleaner interface.
  - we want to be able to download meta data about all datasets withot downloading the full data to be able to explicitly see which scenes are relevant to us (i.e. in terms of having many atek snipptes and gt meshes available)
