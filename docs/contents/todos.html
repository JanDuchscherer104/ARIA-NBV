Action Items & TODOs

1 Overview

This page consolidates all open tasks, questions, and references for the ASE/ATEK + EFM3D NBV project. Nothing has been removed; items are reorganized for clarity.

2 Open Questions & Constraints

  • ASE single trajectory: Each scene has one prerecorded egocentric path; no arbitrary novel viewpoints.
    • For 100 validation scenes, GT meshes exist (EFM3D release) → usable for oracle RRI.
    • For other scenes, EFM3D-predicted meshes act as pseudo-GT.
  • Depth: ASE depth maps are GT.
  • Sparse SLAM PC: Non-GT point clouds are semi-dense; need a strategy to compare candidate-view point clouds against provided SLAM PCs.
  • Action space: Start with discrete candidate selection (VIN-NBV style). Later: continuous pose generation (GenNBV) with collision-free sampling using free-space voxels.
    • Future work: learn to generate continuous poses directly, whose fitness can than be quantified using the RRI predictor; here we can also consider the free space voxels that are used in EFM3D to avoid collisions.

3 Highest Priority

3.1 Metrics

  • Implement OracleRRI with point-cloud–to–mesh distance (Chamfer components).
  • Consider mesh-to-distance-field acceleration and room-level cropping to avoid unrelated geometry - How can we consider only relevant (observable parts) of the scene, such that we don’t have to consider the entire mesh (e.g. there might be different rooms that cannot be considered)?
  • Later: deliver as TorchMetric.

3.2 Candidate View Generation & Sampling

  • implement CandidateViewGenerator and auxiliary mdoules in oracle_rri/oracle_rri/pose_generation, that generated N candidate SE(3) poses (roll = 0) around the latest camera position in the trajectory. Must generate poses of type PoseTW (external/efm3d/efm3d/aria/pose.py)
    • Strategies: uniform over half-sphere with min/max radius & elevation, free-space only, room-bounded (SSL for train, occupancy for eval).
    • Collision check: segment from last pose to candidate must not intersect mesh/voxels.
    • Modular strategy selection via enums/config that allows easy extension and combination of strategies / sampling rules.

3.2.1 Raycasting to Generate Candidate Point Clouds-

  • Implement CandidatePointCloudGenerator and all necessary auxiliary modules in : for each candidate pose, cast rays to GT to render depth map using existing GPU implementations from efm3d.
    • Input: candidate PoseTW, GT mesh.
    • Output: Depth map.
  • Questions:
    • Which camera params should we use for rendering candidate views / projections? RGB camera intrinsics and extrinsics from ASE (see oracle_rri/oracle_rri/data/views.py?)?
    • In which coordinate frame? ego camera for now!
    • Should we use the rendering utilities from EFM3D or some other gpu based renderer (e.g. pyrender, pytorch3d)? This should be implemented in oracle_rri/oracle_rri/rendering. Consider that we have already tried implementing the entire pipeline in a single script. extract the relevant parts and criticise them! panda3d.

3.2.2 Point Cloud Fusion

  • Implement DepthToPointCloud module: for each candidate pose’s depth map, backproject to point cloud `\(\mathcal{P}_{\mathbf{q}}\) in the global frame, using the known camera paramaters from ASE and and then fuse semi-dense SLAM PC \(\mathcal{P}_t\) with candidate-view PC \(\mathcal{P}_{\mathbf{q}}\) → \(\mathcal{P}_{t \cup \mathbf{q}}\).

3.3 Visualization (Streamlit)

  • Interactive app to visualize candidate sampling, PC–mesh distances, and RRI.
  • Efficient caching; quick scene/trajectory switching.
  • should allow for easy adaptation of config (i.e. which scene and snippet to use)
  • all streamlit variables must be fully typed
  • should allow display of the different pipeline stages in different tabs:
    • candidate view sampling (display candidate poses around current camera pose)
    • candidate depth rendering (display rendered depth maps from GT mesh for each candidate pose)
    • candidate point cloud fusion (display fused point clouds for each candidate pose)
    • RRI scoring (display RRI scores for each candidate pose)
  • should have debug print tab where we can easily see the repr prints of the different data structures and logs!

3.4 Data Handling

  • Link ASE ATEK shards and GT meshes to raw ASE; resolve scene_id/snippet_id mapping.
  • Wrap ATEK dataset/downloader to expose:
    • easy ASE ATEK WDS access + GT meshes,
    • metadata file combining scene↔︎snippet mapping,
    • download of metadata without full payload,
    • selection of ATEK snippets based on available GT meshes (ase_mesh_download_urls.json).
  • Download all dataset variants (ATEK, raw ASE, meshes) for scenes with GT.

4 Medium Priority

4.1 Candidate Sampling (learning variants)

  • Lightweight net on top of EVL to predict candidate rays / projections given pose + mesh.

4.2 Evaluation & Benchmarking

  • Surface reconstruction: F-Score vs GT meshes (ATEK + EFM3D utils).
  • Static object detection: CubeRCNN baselines.
  • Reproduce benchmarking scripts in external/ATEK/tools.

4.3 Download & Data Ops

  • Automate downloads using ATEK/Project Aria tools; ensure integrity (sha1).
  • Provide CLI wrappers for scene filters and variant selection (train/val/test).

5 Lower Priority / Backlog

5.1 Pose & Navigation

  • Free-space validation using EFM3D occupancy; collision-aware NBV pathing.

5.2 Misc

  • Extend candidate generation to continuous policy conditioned on fused occupancy / OBB priors.
