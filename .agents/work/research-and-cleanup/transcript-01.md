## The method stack I would actually implement

### Phase 1 — no RL yet: stochastic greedy counterfactual planner

Implement:

[
p(a_i \mid s) = \frac{\exp(score(s,a_i)/\tau)}{\sum_j \exp(score(s,a_j)/\tau)}
]

where `score` is either oracle RRI or predicted RRI. This gives you a **Boltzmann policy** over discrete candidate poses. Use `τ → 0` for greedy behavior, `τ ≈ 1` for exploration, and high `τ` for near-random candidate diversity.

For beam width `B`, use either:

* **independent Boltzmann rollouts**: simple, fast, may duplicate branches;
* **Gumbel-Top-k sampling**: better, samples diverse candidates without replacement;
* **stochastic beam search**: best if you want a principled sequence-level sampler later. ([arXiv][1])

This phase is closest to VIN-NBV, except replacing greedy `argmax` with stochastic sampling. VIN-NBV’s one-step candidate-selection loop is exactly the structural baseline to extend. ([arXiv][2])

### Phase 2 — learn a Q-function from oracle rollouts

Once you generate rollouts, store transitions:

```text
s_t
candidate_set_t
a_t
r_t = oracle_RRI or Δmetric
s_{t+1}
done
valid_action_mask
counterfactual_modalities
```

Then train:

[
Q(s_t,a_t) \leftarrow r_t + \gamma , \mathbb{E}*{a' \sim \text{softmax}(Q(s*{t+1},a')/\tau)} Q(s_{t+1},a')
]

This is a **soft Bellman backup**. The resulting policy is again softmax over Q-values, but now scores represent **multi-step value**, not just immediate RRI. Soft Q-Learning and SAC give the principled maximum-entropy interpretation. ([arXiv][3])

### Phase 3 — protect against offline-RL failure

Because your dataset will be offline and oracle-generated, the danger is Q-value hallucination. The relevant methods are:

* **IQL** if you want a stable offline Q/value method that avoids evaluating unseen actions too aggressively. ([OpenReview][4])
* **CQL** if your learned policy starts exploiting overestimated candidates. ([NeurIPS Papers][5])
* **AWAC** if you later want to fine-tune from offline rollouts with additional online / simulated data. ([arXiv][11])

I would start with **IQL-style value learning** or a simpler masked discrete Double-DQN baseline, then add CQL regularization only if overestimation becomes visible.

They apply this to three parameter groups:

1. **Geometric / photometric Gaussian parameters**

2. **Semantic Gaussian features**

3. **Dynamic deformation network parameters**

## 1. Fisher information: conceptual intuition

Suppose a model has parameters (\theta), and a candidate view (q) would produce an observation (y_q). The model defines a likelihood:

[

> A view has high Fisher information if a small change in the model parameters would noticeably change what the model predicts from that view.

Equivalently, a view is informative if it produces strong gradients with respect to the parameters:

[
g_q = \nabla_\theta \log p(y_q \mid \theta, q)

[

===

\mathbb{E}
\left[
g_q g_q^\top
\right]

So each candidate view gives a matrix saying: “this view constrains these parameter directions.”

## 6. How the deformation network works

The deformation network has two main parts.

### 6.1 Spatial-temporal structure encoder

[

[
(x,y), (x,z), (y,z), (x,t), (y,t), (z,t)

Each plane stores learnable features. For a Gaussian at ((x,y,z)) and time (t), the model queries all relevant planes with bilinear interpolation and fuses the resulting features.

Intuition:

* ((x,y), (x,z), (y,z)) encode spatial structure.
* ((x,t), (y,t), (z,t)) encode how space changes over time.
* Multi-resolution planes capture both coarse and fine motion.

This is much cheaper than storing a full 4D voxel grid.

[

[
\Delta R_i(t) = Head_R(h_i(t))
]

[
\Delta s_i(t) = Head_s(h_i(t))
]

Then the Gaussian is deformed and rendered.

The original 4D-GS paper describes this as an efficient Gaussian deformation field network with a temporal-spatial structure encoder and a tiny multi-head Gaussian deformation decoder. For training labels, store invalid candidates instead of dropping them. This gives you much better data for feasibility learning. Hestia explicitly uses collision-free projection and penalties for unsafe decisions, which is directly relevant here. ([arXiv][1])

# 4. Multi-step rollout policy

For the first thesis implementation, use **stochastic receding-horizon rollouts**, not full RL.

At every step, score all valid candidates:

[
S_i = S(s_t, q_i, e).
]

Convert scores to a Boltzmann policy:

[
\pi_\tau(q_i \mid s_t,e)
========================

\frac{
\exp(S_i/\tau)
}{
\sum_j \exp(S_j/\tau)
}.
]

Here (\tau) is the temperature:

* (\tau \to 0): greedy argmax.
* moderate (\tau): exploration around strong candidates.
* high (\tau): nearly random exploration.

For beam width (B), use **Gumbel-Top-k sampling**:

[
g_i = -\log(-\log u_i), \qquad u_i \sim \mathrm{Uniform}(0,1),
]

[
\tilde{S}_i = \log \pi(q_i \mid s,e) + g_i,
]

then select the top (B) candidates by (\tilde{S}_i). The Gumbel-Top-k / stochastic beam-search paper is highly relevant because it gives exact sampling without replacement and states that the number of model evaluations grows linearly in (k) and maximum sequence length, which matches your desired rollout complexity. ([arXiv][2])

The key design constraint:

[
\text{cost} = O(B \cdot L \cdot N),
]

not

[
O(N^L).
]

That means beam width should mean **B sampled trajectory chains**, not “branch every node into B children.”

# 5. RL formulation after rollout data exists

Once you have counterfactual transitions, define:

[
s_t = {\text{EVL/VIN scene state}, P_t, \text{history}, e, \mathcal{Q}_t}
]

[
a_t = q_t \in \mathcal{Q}_t
]

[
r_t = RRI_e(q_t) - \lambda_{\mathrm{invalid}}\mathbf{1}*{\mathrm{invalid}} - \lambda*{\mathrm{cost}}c(q_t)
]

[
s_{t+1} = T(s_t, q_t)
]

The objective is:

[
J(\pi)
======

\mathbb{E}*{\pi}
\left[
\sum*{t=0}^{L-1}
\gamma^t r_t
\right].
]

For a discrete candidate set, the soft value function is:

[
V(s,e)
======

\alpha
\log
\sum_{q \in \mathcal{Q}(s)}
\exp
\left(
\frac{Q(s,q,e)}{\alpha}
\right),
]

and the soft Bellman target is:

[
y_t
===

r_t
+
\gamma V(s_{t+1},e).
]

The induced policy is:

[
\pi(q \mid s,e)
===============

\frac{
\exp(Q(s,q,e)/\alpha)
}{
\sum_{q'} \exp(Q(s,q',e)/\alpha)
}.
]

This is exactly the bridge between your current softmax rollout sampler and a real learned Q-function. Soft Q-Learning formalizes this maximum-entropy view and expresses the optimal policy as a Boltzmann distribution. ([arXiv][3])

# 6. Most relevant papers

## A. Your core local baseline: VIN-NBV

**Why it matters.** VIN-NBV is the closest paper to your current project: it predicts **Relative Reconstruction Improvement** for candidate views and uses a greedy sequential policy. It explicitly argues that coverage alone is insufficient for complex geometry and occlusions. Your project should keep VIN-NBV as the conceptual baseline and extend it to **entity-conditioned, multi-step rollouts**.

VIN-NBV’s scoring idea is:

[
\widehat{RRI}(q)
================

f_\theta(R_{\mathrm{base}}, C_{\mathrm{base}}, C_q).
]

Your extension should become:

[
\widehat{RRI}_e(q)
==================

f_\theta(s, q, e).
]

Useful ideas to borrow:

* view-conditioned projection features,
* RRI as the primary quality objective,
* candidate ranking rather than full continuous control,
* CORAL / ordinal formulation for skewed RRI labels.

Do **not** keep the one-step greedy decision as the final thesis method. Use it as the baseline.

---

## B. GenNBV

GenNBV is the strongest earlier RL-style NBV baseline for generalizable 5-DoF exploration. It formulates NBV as an MDP, uses a large 5D free-space action space, and combines geometric, semantic, and action-history embeddings. The paper’s motivation is similar to yours: hand-crafted candidate spaces and object-centric setups limit generalization. ([arXiv][4])

Its state embedding is conceptually:

[
s_t = \mathrm{Linear}(s_t^G; s_t^S; s_t^A),
]

where:

* (s_t^G): geometric embedding from probabilistic 3D occupancy grid,
* (s_t^S): semantic/image embedding,
* (s_t^A): historical pose/action embedding.

Useful for you:

* the MDP framing,
* probabilistic occupancy grid as an interpretable state,
* action-history conditioning,
* coverage AUC as an efficiency metric.

Less useful immediately:

* PPO as first method,
* pure coverage reward,
* continuous policy before your oracle/entity-rollout dataset exists.

GenNBV is therefore a **future continuous-action baseline**, not the first implementation target.

---

## C. Hestia / Voxel-Face-Aware Hierarchical NBV

Hestia is extremely relevant to your notes. Its three most transferable ideas are:

1. **Hierarchical action factorization:** first predict a look-at point, then predict the camera position conditioned on that point.
2. **Voxel-face visibility:** treat voxels as cubes with six observable faces, not just occupied points.
3. **Close-greedy training:** use a small discount / immediate reward structure to avoid spurious long-horizon credit assignment. ([arXiv][1])

The key action factorization is:

[
\text{target/look-at point } z_t
================================

g_\phi(s_t),
]

[
\text{camera position } x_t
===========================

\pi_\theta(s_t, z_t).
]

For your thesis, translate this into:

[
z_t = \text{entity center / missing-surface centroid / uncertainty peak},
]

[
q_t = \text{feasible pose looking at } z_t.
]

This is highly compatible with your object-centric plan. Instead of sampling all candidates around the current pose, generate candidates from a mixture:

[
p(q)
====

\lambda_{\mathrm{obj}}p(q \mid \text{look at entity})
+
\lambda_{\mathrm{frontier}}p(q \mid \text{frontier/look-away})
+
\lambda_{\mathrm{explore}}p(q \mid \text{broad shell}).
]

Hestia also motivates your “do not just drop invalid candidates” idea: it shifts predicted actions to nearest collision-free positions and uses penalties for unsafe/invalid actions. ([arXiv][1])

Most important Hestia equation conceptually:

[
r_t
===

## \Delta \text{visible voxel faces}

\text{invalid penalties}.
]

For you:

[
r_t
===

## RRI_e(q_t)

\text{invalid penalties}.
]

Hestia reports strong limited-budget efficiency, for example 92% coverage with 5 acquisitions compared with lower baselines, and attributes this partly to close-greedy training. ([arXiv][1])

## D. FisherRF and Fisher-information NBV

FisherRF reframes active view selection as **expected information gain over model parameters**, not just coverage. Its core idea: select the next view that maximizes Fisher information of the radiance-field parameters. ([arXiv][5])

For a candidate view (q), Fisher information is:

[
F_q(\theta)
===========

\mathbb{E}
\left[
\nabla_\theta \log p(y_q \mid \theta)
\nabla_\theta \log p(y_q \mid \theta)^\top
\right].
]

A cheap score is:

[
S_{\mathrm{FI}}(q)
==================

\mathrm{tr}(F_q).
]

A more principled but expensive score is log-determinant information gain:

[
S_{\mathrm{EIG}}(q)
===================

## \log \det(F_{\mathrm{current}} + F_q)

\log \det(F_{\mathrm{current}}).
]

How to use it for your project:

[
S(q)
====

\hat{RRI}_e(q)
+
\beta \cdot U(q),
]

where (U(q)) could be:

* Fisher score,
* gradient norm,
* entropy over CORAL bins,
* disagreement between geometry and semantic heads.

This is useful because entity-aware RRI can become myopic: a candidate may not immediately improve Chamfer error, but may reduce uncertainty around an object part or unknown surface.

FisherRF explicitly argues that indirect uncertainty heuristics do not necessarily maximize information gain, and it computes observed information directly on radiance-field parameters. ([arXiv][5])

## E. Semantic and Dynamic 3D Gaussian Splatting NBV

The 2025 paper *Next Best View Selections for Semantic and Dynamic 3D Gaussian Splatting* extends Fisher-style active selection to semantic Gaussian parameters and deformation networks. Its key idea is that NBV should improve not only static geometry, but also semantic and temporal/dynamic model parameters. ([arXiv][6])

For your thesis, the transferable score decomposition is:

[
S(q)
====

\lambda_g S_{\mathrm{geom}}(q)
+
\lambda_s S_{\mathrm{sem}}(q)
+
\lambda_d S_{\mathrm{dyn}}(q).
]

In your case:

[
S(q)
====

\lambda_{\mathrm{rri}}\hat{RRI}*e(q)
+
\lambda*{\mathrm{sem}}U_{\mathrm{entity}}(q)
+
\lambda_{\mathrm{feas}}S_{\mathrm{valid}}(q).
]

This paper is less important as a full implementation dependency, but very important as a **conceptual justification** for adding semantic/entity-aware information terms.

## F. OUGS and object-centric 3DGS NBV

OUGS is directly aligned with your object-centric direction. It argues that scene-level uncertainty can be biased by irrelevant background clutter and proposes object-aware uncertainty for 3DGS by propagating Gaussian-parameter covariance through the rendering Jacobian and integrating semantic masks. ([arXiv][7])

Its key transferable idea:

[
S_{\mathrm{object}}(q)
======================

\sum_{i \in \text{object}}
w_i \cdot U_i(q),
]

where (U_i(q)) is uncertainty of scene primitive (i) as viewed from candidate (q).

Your analogous method:

[
S_{\mathrm{object}}(q)
======================

RRI_e(q)
]

or

[
S_{\mathrm{object}}(q)
======================

\hat{RRI}_e(q)
+
\beta U_e(q).
]

The more recent *Informative Object-centric NBV for Object-aware 3DGS* is even closer to your thesis because it explicitly targets object-centric NBV in cluttered scenes and reports that object-centric NBV can improve target-object reconstruction over whole-scene targeting. It uses object-aware 3DGS with instance-level information and confidence-weighted information gain. ([arXiv][8])

Use these papers to justify:

* object-specific objectives,
* instance-aware uncertainty,
* target-conditioned candidate generation,
* background-clutter suppression.

## G. POp-GS and SA-ResGS

POp-GS uses optimal experimental design to quantify information gain in 3D Gaussian Splatting. The relevant point for you is not P-optimality itself, but the idea that 3DGS primitives give an explicit parameter space where uncertainty and view utility can be computed. ([arXiv][9])

SA-ResGS is relevant because it targets sparse-view active reconstruction and tries to stabilize uncertainty estimates with self-augmented point clouds and residual supervision. ([arXiv][10])

These are useful if you decide to use Gaussian splats as a **counterfactual modality generator** or as an auxiliary uncertainty representation. They are less central if your thesis stays strictly on EVL + pointcloud + mesh oracle.

# 7. Architecture papers for VIN v4

## QCNet

QCNet’s useful idea is **query-centric encoding**: instead of re-encoding the whole scene every time the query changes, build reusable invariant scene features and decode relative to each query. The paper explicitly identifies redundant recomputation in sliding-window prediction and introduces query-centric scene encoding for reuse and parallel decoding. ([CVF Open Access][17])

This maps cleanly to VIN v4:

* scene tokens: EVL voxels / points / entities,
* query tokens: candidate poses,
* target tokens: selected OBB/entity,
* history tokens: previous views,
* projection tokens: frustum evidence.

A transformer scorer could be:

[
H =
\mathrm{Transformer}
(
T_{\mathrm{scene}},
T_{\mathrm{entity}},
T_{\mathrm{candidate}},
T_{\mathrm{history}}
)
]

[
\hat{RRI}_e(q_i)
================

\mathrm{MLP}(h_{q_i}).
]

Useful design:

* encode scene once,
* score many candidates in parallel,
* keep candidate-query conditioning explicit.

## LMFormer

LMFormer is not an NBV paper, but the useful idea is **structured-context prioritization** and iterative refinement. It is lane-based trajectory prediction, where the model dynamically prioritizes lanes and uses lane connection information for long-range dependencies. ([arXiv][18])

For your system, replace lanes with:

* frontiers,
* entities,
* regions,
* target object centers,
* expected-RRI centroids.

The analog is:

\mathrm{Attention}(\text{candidate}, \text{entity/portal/frontier tokens}).
]

This is useful for a later semantic-global planner, not for the first implementation.

# 8. Offline RL papers to use after counterfactual rollouts

## IQL

IQL is the best first serious offline RL method for your setting. It avoids evaluating actions outside the dataset, which is crucial when your candidates are generated by a constrained sampler and your oracle is expensive. The paper’s core claim is that it never needs to evaluate actions outside the dataset while still improving over the behavior policy. ([OpenReview][19])

IQL learns:

[
V_\psi(s)
]

by expectile regression:

[
\mathcal{L}_V(\psi)
===================

\mathbb{E}*{(s,a)}
\left[
L_2^\tau
\left(
Q*\theta(s,a) - V_\psi(s)
\right)
\right],
]

then learns:

[
\mathcal{L}_Q(\theta)
=====================

\mathbb{E}
\left[
\left(
Q_\theta(s,a)
-------------

(r + \gamma V_\psi(s'))
\right)^2
\right].
]

Policy extraction is advantage-weighted behavioral cloning:

[
\mathcal{L}_\pi
===============

-\mathbb{E}_{(s,a)}
\left[
\exp(\beta A(s,a))
\log \pi(a \mid s)
\right],
]

where

[
A(s,a)=Q(s,a)-V(s).
]

For your project: train IQL on oracle-generated rollout transitions.

---

## CQL

CQL is useful if learned Q-values become overoptimistic for weird candidates. It explicitly addresses offline RL overestimation under dataset-policy distribution shift and learns a conservative Q-function that lower-bounds policy value. ([NeurIPS Proceedings][20])

A simplified CQL-style penalty is:

[
\mathcal{L}_{\mathrm{CQL}}
==========================

\alpha
\left(
\log \sum_a \exp Q(s,a)
-----------------------

\mathbb{E}_{a \sim \mathcal{D}} Q(s,a)
\right).
]

This pushes down unseen / unsupported action values.

Use this after you observe failure cases like:

* policy selects candidate views far outside training distribution,
* predicted Q is high but oracle RRI is low,
* invalid views receive high predicted value.

---

## Trajectory Transformer

Trajectory Transformer frames offline RL as sequence modeling and uses beam-search-style planning over trajectories. The paper explicitly models distributions over trajectories with a Transformer and repurposes beam search as a planning algorithm. ([arXiv][21])

For your thesis, a future trajectory-token model could represent:

[
(s_t, e, q_t, r_t, s_{t+1})
]

as a sequence:

[
[z_s, z_e, z_q, z_r, z_{s'}].
]

Then planning becomes:

[
\tau^*
======

\arg\max_\tau
\left[
\log p_\theta(\tau)
+
\lambda R(\tau)
\right].
]

This is attractive, but it requires much more rollout data than your first version will have.

---

## Decision Transformer

Decision Transformer conditions action generation on desired return rather than learning a Bellman critic. It casts RL as conditional sequence modeling and generates actions from desired return, states, and previous actions. ([arXiv][22])

For you:

[
\pi(q_t \mid R_{\mathrm{target}}, s_{\le t}, q_{<t}, e)
]

could generate high-RRI trajectories conditioned on desired cumulative RRI.

This is a later-stage method. IQL / masked Q-learning is more direct for the first thesis system.

# 2. Entity-aware RRI: the central contribution

## 2.1 Definition

For target entity (e), with GT or predicted oriented bounding box (B_e), define a margin-expanded box:

[
B_e^{+\delta}.
]

The entity mask is:

[
\chi_e(x)
=========

\mathbf{1}[x \in B_e^{+\delta}].
]

Then:

[
P_t^e = {p \in P_t \mid \chi_e(p)=1},
]

[
P_q^e = {p \in P_q \mid \chi_e(p)=1},
]

[
M_{\mathrm{GT}}^e = M_{\mathrm{GT}} \cap B_e^{+\delta}.
]

The Chamfer-style metric should keep the same two directional components already used in your paper:

[
CD(P^e,M^e)
===========

A(P^e,M^e)+C(P^e,M^e),
]

where

[
A(P^e,M^e)
==========

\frac{1}{|P^e|}
\sum_{p\in P^e}
\min_{f\in F_e}
d(p,f)^2
]

is an **accuracy** term, and

[
C(P^e,M^e)
==========

\frac{1}{|F_e|}
\sum_{f\in F_e}
\min_{p\in P^e}
d(p,f)^2
]

is a **completeness** term.

For NBV, completeness is often more important than accuracy: an object may be accurately reconstructed on its visible side while still missing its back side. This is exactly where entity-aware NBV becomes useful.

## 2.2 Why this is better than scene-level RRI

Scene-level RRI can choose views that improve walls, floor, or irrelevant clutter. Entity-level RRI asks:

> Which next view improves the object I care about?

This aligns with OUGS and object-centric Gaussian-splatting NBV papers. OUGS argues that scene-level uncertainty is often biased by background clutter and proposes object-aware uncertainty for 3DGS by propagating Gaussian parameter covariance through rendering Jacobians. ([arXiv][3]) The 2026 object-centric NBV paper explicitly targets cluttered scenes and reports that focusing on a target object can reduce target-object depth error beyond whole-scene NBV. ([arXiv][4])

For your thesis, use these papers as the object-centric motivation, but implement the simpler and more rigorous version first: **OBB-cropped oracle RRI**.

# 3. Invalid candidates: use a separate feasibility head, not a lowest CORAL bin

Your notes ask whether invalid views should become the lowest CORAL class. I would avoid that.

Invalidity is not ordinal. A low-RRI valid candidate and a collision candidate are semantically different. Treat them separately:

[
\hat{RRI}*e(q) = f*\theta^{\mathrm{rri}}(s,q,e),
]

[
\hat{p}*{\mathrm{valid}}(q) = f*\theta^{\mathrm{valid}}(s,q,e),
]

[
\hat{c}*{\mathrm{reason}}(q) = f*\theta^{\mathrm{reason}}(s,q,e).
]

Then define the planner score:

[
S(s,q,e)
========

## \hat{RRI}_e(q)

## \lambda_{\mathrm{invalid}}\left(1-\hat{p}_{\mathrm{valid}}(q)\right)

## \lambda_{\mathrm{collision}}\hat{p}_{\mathrm{collision}}(q)

\lambda_{\mathrm{motion}}c_{\mathrm{motion}}(q).
]

Supervision:

[
\mathcal{L}
===========

\mathcal{L}*{\mathrm{CORAL}}
+
\lambda*{\mathrm{reg}}\mathcal{L}*{\mathrm{Huber}}
+
\lambda*{\mathrm{valid}}\mathcal{L}*{\mathrm{BCE}}
+
\lambda*{\mathrm{reason}}\mathcal{L}_{\mathrm{CE}}.
]

This connects nicely to Hestia, which uses explicit invalid-action penalties and also projects predicted actions toward collision-free positions rather than relying only on reward shaping. Hestia defines invalid/collision terms in the reward and shifts the predicted position to a nearest collision-free point for final capture. ([arXiv][5])

# 4. Candidate generation: mix object-centric and exploration candidates

Your current Aria-NBV pipeline samples candidate camera poses around the current rig pose. For entity-aware planning, I would use a mixture distribution:

[
p(q\mid s,e)
============

\lambda_1 p_{\mathrm{lookat}}(q\mid e)
+
\lambda_2 p_{\mathrm{default}}(q\mid s)
+
\lambda_3 p_{\mathrm{frontier}}(q\mid s,e).
]

## 4.1 Object look-at candidates

Sample camera centers around the target entity:

[
c_q = z_e + r u,
\qquad
u\sim \mathcal{S}^2,
\qquad
r\sim U(r_{\min},r_{\max}),
]

where (z_e) is either the OBB center or a missing-surface centroid:

[
z_e
===

\frac{
\sum_{x\in \mathrm{missing}(e)} w_x x
}{
\sum_x w_x
}.
]

Orient the camera toward (z_e):

[
R_q = \mathrm{LookAt}(c_q,z_e)\circ R_{\mathrm{jitter}}.
]

This is the direct analog of Hestia’s hierarchical policy: first predict a look-at point, then predict the camera position conditioned on that point. Hestia explicitly introduces this decomposition to reduce the difficulty of the continuous 5-DoF action space. ([arXiv][5])

## 4.2 Default look-away candidates

Keep some candidates from the current pose-based sampler. This prevents overfitting to already-visible objects and helps discover portals, new rooms, and non-target geometry.

## 4.3 Frontier / missing-surface candidates

Generate candidates that see currently missing entity surfaces. A simple version uses normals or mesh-surface deficit:

[
x^\star
=======

\arg\max_x
\left[
\text{missingness}(x)\cdot \text{visibility}(x,q)
\right].
]

This is where Hestia’s face-aware idea becomes useful. It treats each voxel as a cube with six faces, not as a point, because point-only visibility can miss directional coverage. ([arXiv][5]) For Aria-NBV, you can implement a cheaper variant:

```text
for each point/voxel/entity surface patch:
    store observed direction histogram
    store normal-aligned coverage
    store incidence-angle statistics
```

A candidate that observes a missing normal direction receives higher value.

# 5. Multi-step planning: stochastic rollout first, Q-learning later

## 5.1 Stochastic softmax rollout

For each candidate (q_i), compute a score:

[
S_i = S(s_t,q_i,e).
]

Then sample via a Boltzmann / softmax policy:

[
\pi_\tau(q_i\mid s_t,e)
=======================

\frac{\exp(S_i/\tau)}
{\sum_j \exp(S_j/\tau)}.
]

This gives you a direct bridge from single-step RRI ranking to RL-style stochastic control. Soft Q-Learning formalizes this idea: the maximum-entropy optimal policy can be expressed as a Boltzmann distribution over Q-values. ([arXiv][6])

## 5.2 Beam width via Gumbel-Top-k

For (B) rollout trajectories, use Gumbel-Top-k instead of repeatedly sampling from softmax and risking duplicates:

[
g_i=-\log(-\log u_i),
\qquad
u_i\sim U(0,1),
]

[
\tilde{S}_i = \log \pi(q_i\mid s,e)+g_i.
]

Select the top (B). The Gumbel-Top-k / stochastic beam-search paper shows how to sample (k) sequences without replacement and notes that model evaluations grow linearly in (k) and maximum sequence length, which matches your desired (O(BLN)) complexity. ([arXiv][7])

Your rollout should be:

[
O(B\cdot L\cdot N),
]

not:

[
O(N^L).
]

So **beam width means B sampled chains**, not exponential branching.

# 6. RL formulation

Once you have counterfactual rollouts, define an MDP.

## 6.1 State

[
s_t =
{
P_t,,
F_{\mathrm{EVL}},,
h_{\mathrm{history}},,
h_e,,
\mathcal{Q}*t,,
m*{\mathrm{valid}}
}.
]

Components:

* (P_t): current reconstructed points.
* (F_{\mathrm{EVL}}): EVL voxel / occupancy / object features.
* (h_{\mathrm{history}}): past selected views.
* (h_e): encoded target entity OBB/class/confidence.
* (\mathcal{Q}_t): candidate poses.
* (m_{\mathrm{valid}}): validity/collision mask.

The EVL stack is suitable because the official release provides pretrained EVL weights, datasets for ASE/AEO/ADT, distributed EVL training code, and ATEK integration. ([GitHub][8])

## 6.2 Action

Discrete candidate version:

[
a_t = q_i \in \mathcal{Q}_t.
]

Continuous version:

[
a_t = (\Delta x,\Delta y,\Delta z,\Delta \mathrm{yaw},\Delta \mathrm{pitch}).
]

For this thesis, start with discrete candidates. Continuous 5-DoF comes later.

## 6.3 Reward

Entity RRI reward:

[
r_t
===

## RRI_e(q_t)

## \lambda_{\mathrm{invalid}}\mathbf{1}_{\mathrm{invalid}}

## \lambda_{\mathrm{motion}}c(q_t)

\lambda_{\mathrm{turn}}|\Delta\theta|.
]

For close-greedy training, set (\gamma) low:

[
\gamma \in [0.1,0.5].
]

Hestia uses a small discount factor to prioritize immediate improvements and reduce spurious correlations from assigning large future rewards to bad current actions. ([arXiv][5]) This is highly relevant to your “trespasses to new areas” question: use low-discount local RL first, then later add semantic/global planning for long-horizon room transitions.

## 6.4 Soft Q target

For a discrete candidate set:

[
V(s_{t+1},e)
============

\alpha
\log
\sum_{q'\in\mathcal{Q}*{t+1}}
\exp(Q(s*{t+1},q',e)/\alpha).
]

Then:

[
y_t
===

r_t+\gamma V(s_{t+1},e).
]

Critic loss:

[
\mathcal{L}_Q
=============

\left(Q_\theta(s_t,q_t,e)-y_t\right)^2.
]

Policy:

[
\pi(q\mid s,e)
==============

\frac{
\exp(Q(s,q,e)/\alpha)
}{
\sum_{q'}\exp(Q(s,q',e)/\alpha)
}.
]

This is the natural next step after your stochastic rollouts.

# 7. RL papers most relevant to your system

## 7.1 Hestia

Hestia is the most important RL-NBV paper for your next design phase. It uses a 5-DoF viewpoint policy, hierarchical look-at / position prediction, face-aware voxel observations, invalid-action penalties, and close-greedy training. It is also trained in NVIDIA IsaacLab with 256 scenes in parallel. ([arXiv][5])

Transferable ideas:

* track directional surface observability, not just occupancy;
* use immediate reward / low discount before long-horizon RL;

## 7.2 GenNBV

GenNBV is the canonical continuous 5-DoF NBV RL baseline. It formulates NBV as an MDP, uses RGB-D observations, pose history, and a probabilistic occupancy grid, and trains PPO over a 5D free-space action space. It reports strong cross-dataset coverage results, but its reward is coverage gain rather than reconstruction-quality RRI.

Use it for:

* MDP definition;
* 5-DoF pose/action-space conventions;
* occupancy-grid state updates;
* continuous-action baseline.

Do not use PPO first. It is too expensive if your reward depends on oracle RRI.

## 7.3 VIN-NBV

VIN-NBV is the closest conceptual baseline to Aria-NBV. It predicts RRI for sampled candidate views and uses a greedy sequential policy. It also uses CORAL because direct RRI regression is brittle under skewed, stage-dependent RRI distributions.

Use it for:

* RRI as quality objective;
* candidate ranking;
* ordinal labels;
* greedy baseline;
* reconstruction-quality rather than coverage.

Your contribution is to extend this to:

[
\widehat{RRI}_e(s,q)
\quad\rightarrow\quad
\widehat{Q}_e(s,q)
]

and from one-step to multi-step rollouts.

## 7.4 Soft Q-Learning

Soft Q-Learning gives the mathematical foundation for your temperature-softmax policy. It expresses the optimal maximum-entropy policy as a Boltzmann distribution over Q-values. ([arXiv][6])

Use it when you want:

[
\pi(q\mid s)\propto \exp(Q(s,q)/\alpha).
]

This makes your temperature parameter principled rather than heuristic.

## 7.5 IQL

IQL is the best offline RL method to try first after you create rollout data. Its key property is that it avoids evaluating actions outside the dataset while still performing multi-step dynamic programming; this matters because your candidate distribution is constrained and oracle labels are expensive. ([OpenReview][9])

The value loss:

[
\mathcal{L}_V(\psi)
===================

\mathbb{E}*{(s,a)\sim D}
\left[
L_2^\tau(Q*\theta(s,a)-V_\psi(s))
\right],
]

where (L_2^\tau) is asymmetric expectile regression.

Then:

[
\mathcal{L}_Q(\theta)
=====================

\mathbb{E}
\left[
Q_\theta(s,a)
-------------

(r+\gamma V_\psi(s'))
\right]^2.
]

Policy extraction:

[
\mathcal{L}_\pi
===============

-\mathbb{E}_{(s,a)\sim D}
\left[
\exp(\beta A(s,a))\log\pi(a\mid s)
\right].
]

For Aria-NBV, use IQL after you have a static dataset of oracle rollouts.

## 7.6 CQL

CQL is relevant if the learned Q-function starts assigning high value to unsupported or invalid candidate poses. CQL learns conservative Q-values so the expected policy value lower-bounds the true value, reducing offline-RL overestimation under distribution shift. ([NeurIPS Proceedings][10])

Simplified CQL penalty:

[
\mathcal{L}_{\mathrm{CQL}}
==========================

\alpha
\left[
\log\sum_a \exp Q(s,a)
----------------------

\mathbb{E}_{a\sim D}Q(s,a)
\right].
]

Use it when:

* predicted Q is high but oracle RRI is low;
* invalid candidates get high value;
* policy exploits holes in the learned scorer.

## 7.7 PPO and SAC

PPO is useful for an online simulator because it alternates between collecting interaction data and optimizing a clipped surrogate objective; it is simple and robust but sample-hungry. ([arXiv][11]) SAC is more suitable for continuous actions because it is off-policy and maximum-entropy, seeking high reward while maintaining stochasticity. ([arXiv][12])

Use PPO/SAC only after you have a fast simulator. For immediate research, use stochastic rollouts + IQL/CQL.

## 7.8 Trajectory Transformer and Decision Transformer

Trajectory Transformer frames offline RL as sequence modeling and explicitly uses beam-search-style planning over trajectories. ([arXiv][13]) Decision Transformer conditions actions on desired return, past states, and previous actions instead of learning a Bellman critic. ([arXiv][14])

These are useful later, once your rollout dataset is large. A trajectory token could be:

```text
[state embedding, entity token, candidate pose token, RRI bin, validity token]
```

Then trajectory generation becomes sequence decoding.

# 8. Information gain / Fisher papers

## 8.1 FisherRF

FisherRF is important because it moves from output-space metrics to parameter-space information. It selects views by maximizing Fisher Information / Expected Information Gain over radiance-field parameters. ([arXiv][15])

Fisher matrix for candidate (q):

[
F_q(\theta)
===========

\mathbb{E}
[
\nabla_\theta \log p(y_q\mid \theta)
\nabla_\theta \log p(y_q\mid \theta)^\top
].
]

Cheap score:

[
S_{\mathrm{FI}}(q)=\mathrm{tr}(F_q).
]

For Aria-NBV, use Fisher/uncertainty as an auxiliary exploration term:

[
S(q)
====

\hat{RRI}_e(q)
+
\beta U(q).
]

Where (U(q)) could be:

* entropy over CORAL bins;
* gradient norm of the RRI head;
* Fisher trace approximation;
* ensemble variance;
* disagreement between geometry and semantic heads.

## 8.2 Semantic / dynamic 3DGS NBV

The semantic/dynamic 3DGS NBV paper extends Fisher-style NBV to semantic Gaussian features and deformation networks. It explicitly decomposes informativeness into geometric, semantic, and dynamic/deformation components. ([arXiv][16])

The useful abstraction:

[
S(q)
====

\lambda_g S_{\mathrm{geom}}(q)
+
\lambda_s S_{\mathrm{sem}}(q)
+
\lambda_d S_{\mathrm{dyn}}(q).
]

For your thesis:

[
S(q)
====

\lambda_{\mathrm{rri}}\hat{RRI}*e(q)
+
\lambda*{\mathrm{sem}}U_e(q)
+
\lambda_{\mathrm{valid}}S_{\mathrm{valid}}(q).
]

Do not implement dynamic 3DGS now. Use the paper to justify semantic/entity-aware scoring and uncertainty bonuses.

## 8.3 POp-GS

POp-GS uses optimal experimental design for 3DGS and proposes ways to quantify uncertainty / information gain in 3D Gaussian Splatting, including P-optimality and approximations over Gaussian uncertainty. ([arXiv][17])

Use it if you later train splats per ASE scene and want a Gaussian-based uncertainty baseline.

## 8.4 ActiveGS / ActiveSplat

ActiveGS and ActiveSplat are relevant if you want an online mapping system that updates a Gaussian representation and plans views from it. ActiveSplat explicitly describes a unified framework for online mapping, viewpoint selection, and path planning using Gaussian splatting. ([IPB Bonn][18])

These are useful for future counterfactual RGB / photorealistic rollouts, not for the first entity-RRI implementation.

# 10. Representation / architecture papers

## 10.1 QCNet

QCNet introduces query-centric scene encoding to avoid redundant re-encoding when queries change. ([CVF Open Access][22]) This is highly relevant to VIN v4.

Your scorer should encode the scene once:

[
T_{\mathrm{scene}} = E_{\mathrm{scene}}(F_{\mathrm{EVL}}, P_t)
]

then score many query candidates:

[
h_i = \mathrm{CrossAttention}(T_{q_i}, T_{\mathrm{scene}}, T_e).
]

This is exactly a query-centric candidate scorer.

## 10.2 LMFormer

LMFormer uses lane-aware transformer modeling and iterative refinement with modest added computation. ([CVF Open Access][23])

The transferable idea is not lanes, but **structured candidate support**. Replace lanes by:

```text
portals
frontiers
entities
missing-surface clusters
uncertainty peaks
```

Then use iterative refinement:

[
q^{(k+1)} = q^{(k)} + \Delta q^{(k)}.
]

## 10.3 CORAL

CORAL is already appropriate for skewed RRI targets. It transforms ordinal prediction into (K-1) binary threshold tasks and guarantees rank-consistent logits. ([arXiv][24])

For (K) RRI bins:

[
t_k = \mathbf{1}[y>k],
\qquad
k=0,\dots,K-2.
]

Loss:

[
\mathcal{L}_{\mathrm{CORAL}}
============================

-\sum_{k=0}^{K-2}
\left[
t_k\log p_k+(1-t_k)\log(1-p_k)
\right].
]

Keep CORAL for valid candidates. Add a separate validity head.

## 10.4 Learnable Fourier Features

Learnable Fourier Features are useful for encoding candidate poses, voxel positions, entity centers, and surface-deficit points. The paper proposes trainable Fourier mappings for multi-dimensional spatial coordinates in transformer-like models. ([arXiv][25])

Use for:

[
\phi(x)
=======

\mathrm{MLP}
\left[
\sin(Bx), \cos(Bx)
\right].
]

Inputs:

```text
candidate translation
candidate rotation-6D
entity OBB center
voxel center
surface point
viewing direction
```

# 11. Simulator survey

## 11.1 Native ASE simulator

Best match conceptually, but I did not find a public interactive ASE generator/simulator. Public docs describe ASE as a downloadable 100k sequence dataset with data access APIs, 23 TB total download scale, and tooling for download/visualization. ([Facebook Research][2])

## 11.2 Custom Aria-NBV Gym with PyTorch3D

This should be your first simulator.

Why:

* perfectly aligned with your current oracle RRI;
* no dependency on external physics;
* exact GT mesh reward;
* exact entity OBB cropping;
* cheap enough for discrete rollouts;
* easiest to debug.

Step:

```python
obs = env.reset(scene_id, anchor_pose, target_entity)
for t in range(L):
    candidates = env.sample_candidates(state, target_entity)
    action = policy(candidates)
    obs, reward, done, info = env.step(action)
```

Transition:

```text
render candidate depth from GT mesh
backproject to point cloud
fuse with P_t
update entity coverage / visibility
compute RRI_e
return next state
```

Use PyTorch3D for rendering and point/mesh geometry. PyTorch3D’s renderer is modular, differentiable, and implemented in PyTorch/C++/CUDA, which fits your current GPU batched oracle pipeline. ([PyTorch3D][26])

This simulator is enough for:

* stochastic rollout generation;
* offline RL transitions;
* masked Q-learning;
* IQL/CQL;
* ablations over candidate generation;
* entity-aware reward design.

Limitation: only 100 GT mesh scenes unless more GT/pseudo-GT is added.

## 11.3 Habitat-Sim / Habitat-Lab

Habitat is the best external simulator for **fast indoor egocentric navigation / view planning**.

Why:

* configurable agents and multiple sensors;
* built-in support for Matterport3D, Gibson, Replica, HM3D, etc.;
* very high rendering throughput;
* Habitat-Lab supports RL task definitions and baselines. ([AI Habitat][27])

Habitat-Sim docs report thousands of FPS on Matterport3D and over 10,000 FPS multi-process on a single GPU. ([AI Habitat][27]) Habitat-Lab is a high-level library for embodied AI tasks and includes RL baselines such as PPO/DD-PPO. ([AI Habitat][28])

Datasets:

* HM3D: 1000 building-scale textured 3D reconstructions. ([arXiv][29])
* HM3D-Sem: dense semantic annotations over 216 spaces and 142k+ object instances. ([AI Habitat][30])
* Replica: 18 highly photorealistic indoor meshes with HDR textures and semantic/instance information. ([arXiv][31])
* ReplicaCAD: interactive apartment dataset for Habitat tasks. ([AI Habitat][32])

Recommendation: **use Habitat as the first external online RL simulator** if your online training does not require accurate drone physics.

Use Habitat for:

```text
object-centric view planning
semantic target selection
portal/frontier exploration
large-scale online RL
simulated RGB-D/semantic observations
```

## 11.4 Isaac Sim / Isaac Lab

Isaac is best if you need **robot/drone embodiment, physics, realistic sensors, and sim-to-real motion**.

Isaac Sim is an open-source Omniverse-based robotics simulation framework for testing and synthetic data generation in physically based virtual environments. It supports physics, sensors, ROS/ROS2, synthetic data, and can ingest URDF/CAD/USD assets. ([NVIDIA Developer][33]) Isaac Lab is the GPU-accelerated robot-learning layer built on Isaac Sim, supporting reinforcement learning, imitation learning, motion planning, RGB/depth/segmentation cameras, IMUs, LiDAR, and other sensors. ([GitHub][34])

Hestia uses NVIDIA IsaacLab to randomly simulate 256 scenes in parallel with a Crazyflie drone and RGB-D camera. ([arXiv][5])

Recommendation: **use Isaac Lab if you commit to continuous 5-DoF / drone-like RL.** It is heavier than Habitat but closer to GenNBV/Hestia-style continuous control.

Use Isaac for:

```text
continuous 5-DoF control
drone pose constraints
collision-free trajectory realization
physics-aware rollout
sim-to-real robotics story
```

## 11.5 ProcTHOR / AI2-THOR

ProcTHOR procedurally generates diverse, interactive, customizable 3D environments and is designed to scale embodied AI training by orders of magnitude. ([procthor.allenai.org][35]) AI2-THOR supports egocentric RGB, instance segmentation, semantic segmentation, depth, normals, top-down views, and adjustable camera properties. ([GitHub][36])

Recommendation: use ProcTHOR if you want **massive object/house diversity** and semantic embodied tasks, but it is less aligned with Aria/EVL geometry than Habitat or your custom PyTorch3D environment.

## 11.6 iGibson

iGibson provides fast rendering and physics, interactive objects, hundreds of reconstructed scenes, 12k+ additional compatible scenes, domain randomization, motion planners, and demonstration tools. ([svl.stanford.edu][37])

Recommendation: good for household interaction and object manipulation, less ideal than Habitat/Isaac for your first NBV training loop.

## 11.7 ManiSkill / SAPIEN

ManiSkill 3 is a GPU-parallelized robot simulation and training framework powered by SAPIEN, with RGB-D/segmentation collection reportedly reaching 30,000+ FPS on high-end hardware, plus many RL/IL/VLA baselines. ([maniskill.readthedocs.io][38])

Recommendation: excellent for manipulation and object-centric tasks, but not the first choice for room-scale egocentric NBV.

## 11.8 Genesis

Genesis is a new general-purpose robotics/embodied-AI simulation platform with fast physics and photorealistic rendering ambitions. ([genesis-world.readthedocs.io][39])

Recommendation: promising but too new/risky for a master’s thesis core dependency. Track it; do not base the thesis on it yet.

# Roadmap through September 2026

## Milestone 0 — Repo stabilization and scope lock

**Dates:** 27 Apr – 12 May
**Goal:** Make the repo clean enough that new thesis work does not pile onto stale PRs, stale docs, or unstable cache/device behavior.

Main outcome: **one canonical thesis direction, clean PR queue, stable training/cache/dev workflow.**

## Milestone 1 — Data scale, cache contracts, and oracle reliability

**Dates:** 13 May – 31 May
**Goal:** Make the oracle/cache pipeline scalable and reliable enough for entity-aware labels and multi-step rollouts.

Main outcome: **reproducible cache generation over more mesh-supervised ASE snippets, no fragile downsampling, no silent cache contract drift.**

## Milestone 2 — Entity-aware oracle RRI

**Dates:** 1 Jun – 23 Jun
**Goal:** Implement the thesis core: object/entity-specific RRI using GT OBBs and mesh/point cropping.

Main outcome: **per-entity oracle labels: `RRI_e`, `Δaccuracy_e`, `Δcompleteness_e`, validity labels, selected entity metadata.**

## Milestone 3 — Object-centric candidate generation and feasibility

**Dates:** 24 Jun – 14 Jul
**Goal:** Generate candidate views that are useful for a target object, not only a generic shell around the current rig pose.

Main outcome: **mixed candidate sampler: default look-away + object look-at + missing-surface/frontier candidates, with validity penalties instead of only hard dropping.**

## Milestone 4 — Entity-conditioned VIN scorer

**Dates:** 15 Jul – 4 Aug
**Goal:** Turn VIN from a scene-level candidate scorer into a target-conditioned scorer.

Main outcome: **VIN predicts `RRI_e(q)` and validity from `(scene, entity, candidate)` inputs.**

## Milestone 5 — Multi-step counterfactual rollout

**Dates:** 5 Aug – 25 Aug
**Goal:** Add bounded-complexity multi-step planning over discrete candidates.

Main outcome: **stochastic greedy / Gumbel-Top-k rollouts with complexity `O(B · L · N)`, cumulative entity RRI metrics, rollout cache.**

## Milestone 6 — Offline RL / custom simulator baseline

**Dates:** 26 Aug – 12 Sep
**Goal:** Convert rollouts into an MDP dataset and train first value/policy baselines.

Main outcome: **Aria-NBV Gym environment, masked discrete Q / soft-Q baseline, optional IQL/CQL prototype.**

## Milestone 7 — Experiments, ablations, and thesis write-up

**Dates:** 13 Sep – 30 Sep
**Goal:** Lock evidence, produce figures, update paper/docs, and prepare final thesis narrative.

Main outcome: **one-step vs multi-step, scene-level vs entity-level, oracle vs learned, candidate-generator ablations, final paper figures.**

Schould we use rerun for plotting?

[Agent-Turn18]

Yes — **use Rerun, but only for interactive 3D/temporal diagnostics, not as the general plotting backend.**

For ARIA-NBV, Rerun is a very good fit for **rollout visualization**: candidate frusta, selected trajectories, point clouds, OBBs, meshes, RGB/depth frames, and per-step scalar timelines. It supports 3D point clouds (`Points3D`), line strips, meshes, camera/pinhole models, transforms, blueprints/layouts, time-indexed recordings, partial updates, and `.rrd` files for offline sharing. ([Rerun][1])

## Recommendation

Use this split:

| Use case                                                 | Tool                                                                               |
| -------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| Interactive 3D rollout debugging                         | **Rerun**                                                                          |
| Candidate frusta / point clouds / trajectories over time | **Rerun**                                                                          |
| Streamlit dashboard controls and tables                  | **Streamlit**                                                                      |
| Final thesis plots, ablation tables, metric curves       | **Matplotlib / Plotly**                                                            |
| Paper-ready static geometry figures                      | Plotly export, PyTorch3D/Open3D screenshots, or manually curated Rerun screenshots |
| W&B scalar tracking                                      | W&B                                                                                |

So: **Rerun should replace ad-hoc 3D plotting, not Matplotlib, Plotly, or Streamlit.**

## Why Rerun fits ARIA-NBV

Your diagnostics are inherently **spatiotemporal**:

```text
snippet → candidate set → selected pose → counterfactual point cloud → updated reconstruction → next step
```

That is exactly where Rerun shines. A `.rrd` recording can store an entire rollout and later be opened with `rerun path/to/file.rrd`; Rerun’s docs explicitly support saving logs to `.rrd` for offline analysis and sharing. ([Rerun][2])

A Rerun rollout recording should show:

```text
/world/mesh_gt
/world/semidense_points_t
/world/entity_gt_obb
/world/candidates/step_000/frusta
/world/candidates/step_000/scores
/world/selected_trajectory
/world/candidate_points/selected
/world/rri/entity
/world/rri/cumulative
/world/images/rgb
/world/images/depth
```

Then you scrub the timeline by `rollout_step`.

## What to log first

Implement one small optional module:

```text
aria_nbv/diagnostics/rerun_logger.py
```

with functions:

```python
log_scene_mesh(...)
log_semidense_points(...)
log_entity_obb(...)
log_candidate_frusta(...)
log_selected_candidate(...)
log_rollout_trajectory(...)
log_candidate_pointcloud(...)
log_rri_scalars(...)
log_rgb_depth(...)
```

Start with only these archetypes:

* `Points3D` for semi-dense and candidate point clouds;
* `LineStrips3D` for trajectories and frusta;
* `Mesh3D` for GT/cropped meshes;
* `Pinhole` / camera transforms for candidate cameras;
* `Scalars` for RRI, cumulative RRI, entropy, validity fraction;
* `Image` / depth image if needed.

Rerun’s archetype list includes `Mesh3D`, `Points3D`, `LineStrips3D`, and camera-related archetypes; it also supports transform hierarchies, which is important for world/rig/camera frames. ([Rerun][3])

## Important caveat: coordinate frames

Be careful: Rerun camera examples/docs commonly use an RDF convention for camera coordinates, where the camera frustum points along the configured forward axis and the default forward for RDF is `+Z`; the docs also note the camera up direction follows the chosen view-coordinate convention. ([ref.rerun.io][4])

Your Aria stack uses **LUF** camera frames. So do not “just log poses” without a conversion layer.

Add one explicit function:

```python
T_rerun_cam_world = aria_luf_to_rerun_rdf(T_aria_cam_world)
```

and test it with:

```text
1. candidate frustum points toward expected target
2. image plane is not mirrored
3. depth point cloud aligns with mesh
4. selected trajectory appears in correct world frame
```

This avoids repeating the CW90 / pose-convention pain.

## How to integrate with Streamlit

Do **not** try to make Rerun the Streamlit plotting engine at first.

Use this workflow:

```text
Streamlit = controls, tables, scalar summaries
Rerun = external 3D viewer opened from .rrd recording
```

Add a Streamlit button:

```text
"Export selected rollout to Rerun .rrd"
```

Then show:

```bash
rerun outputs/rerun/rollout_<id>.rrd
```

Rerun can be embedded into web apps using an iframe or JavaScript package, but that is extra complexity and not necessary for the thesis prototype. ([Rerun][5])

## What not to do

Do **not**:

* add Rerun as a hard core dependency;
* rewrite all plotting around Rerun;
* use Rerun for final ablation plots;
* bury Rerun logging inside core RRI/math code;
* make Streamlit depend on Rerun viewer state;
* log huge full-resolution artifacts by default.

Make it optional:

```toml
[project.optional-dependencies]
viz = ["rerun-sdk"]
```

or:

```toml
[project.optional-dependencies]
diagnostics = ["rerun-sdk"]
```

Rerun’s Python SDK can be installed via `pip install rerun-sdk`, and the Python SDK includes the Viewer binary according to the project’s getting-started docs. ([GitHub][6])
