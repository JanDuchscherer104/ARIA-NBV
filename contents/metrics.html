1 Reconstruction Quality Metrics & RRI Computation

This document provides a comprehensive overview of reconstruction quality metrics and how to compute the Relative Reconstruction Improvement (RRI) score for Next-Best-View planning.

1.1 Surface Reconstruction Metrics

1.1.1 Overview

The evaluation of 3D reconstruction quality relies on comparing predicted reconstructions against ground truth meshes. Both ATEK and EFM3D provide implementations for computing these metrics using:

  • Trimesh: For mesh loading and surface sampling
  • PyTorch: For efficient distance computations
  • Point-to-mesh distance functions: Core geometric primitives

1.1.2 Key Metrics

1.1.2.1 1. Accuracy (Prediction → Ground Truth)

Measures how close predicted points are to the ground truth surface.

\[ \text{Accuracy} = \frac{1}{|\mathcal{P}|} \sum_{\mathbf{p} \in \mathcal{P}} \min_{\mathbf{q} \in \mathcal{M}_{\text{GT}}} \|\mathbf{p} - \mathbf{q}\|_2 \]

  • \(\mathcal{P}\): Sampled points from predicted mesh/reconstruction
  • \(\mathcal{M}_{\text{GT}}\): Sampled points from ground truth mesh
  • Lower values indicate better reconstruction accuracy

1.1.2.2 2. Completeness (Ground Truth → Prediction)

Measures how well the prediction covers the ground truth surface.

\[ \text{Completeness} = \frac{1}{|\mathcal{M}_{\text{GT}}|} \sum_{\mathbf{q} \in \mathcal{M}_{\text{GT}}} \min_{\mathbf{p} \in \mathcal{P}} \|\mathbf{p} - \mathbf{q}\|_2 \]

  • Lower values indicate better coverage of the ground truth

1.1.2.3 3. Chamfer Distance (Bidirectional)

Combines both accuracy and completeness:

\[ \text{CD}(\mathcal{P}, \mathcal{M}_{\text{GT}}) = \text{Accuracy} + \text{Completeness} \]

This symmetric metric captures both over-reconstruction (accuracy) and under-reconstruction (completeness).

1.1.2.4 4. Precision, Recall & F-score

Binary classification metrics at threshold \(\tau\) (typically 5cm):

\[ \text{Precision}_\tau = \frac{|\{\mathbf{p} \in \mathcal{P} : \min_{\mathbf{q} \in \mathcal{M}_{\text{GT}}} \|\mathbf{p} - \mathbf{q}\| < \tau\}|}{|\mathcal{P}|} \]

\[ \text{Recall}_\tau = \frac{|\{\mathbf{q} \in \mathcal{M}_{\text{GT}}: \min_{\mathbf{p} \in \mathcal{P}} \|\mathbf{p} - \mathbf{q}\| < \tau\}|}{|\mathcal{M}_{\text{GT}}|} \]

\[ \text{F-score}_\tau = \frac{2 \cdot \text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}} \]

1.3 ATEK Implementation

1.3.1 evaluate_single_mesh_pair()

Located in atek/evaluation/surface_reconstruction/surface_reconstruction_metrics.py.

Purpose: Complete evaluation pipeline that computes all reconstruction quality metrics between a predicted mesh and ground truth mesh.

Workflow:

  1. Load meshes: Uses trimesh.load() to load both predicted and GT meshes
  2. (Optional) Gravity correction: For ADT datasets, aligns gravity direction via correct_adt_mesh_gravity()
  3. (Optional) Height slicing: Cuts meshes at specified height using plane slicing
  4. Sample surfaces: Uniformly samples sample_num points from both meshes using trimesh.sample.sample_surface()
    • Seeds RNG for reproducibility
    • Poisson disk sampling ensures uniform distribution
  5. Compute bidirectional distances: Calls compute_pts_to_mesh_dist() twice (see below for details)
  6. Calculate metrics: Computes accuracy, completeness, precision@5cm, recall@5cm, F-score@5cm

Parameters:

  • pred_mesh_filename: Path to predicted mesh (.ply, .obj, etc.)
  • gt_mesh_filename: Path to ground truth mesh
  • sample_num: Number of points to sample (default: 10,000)
  • step: Batch size for distance computation (default: 50,000 triangles)
  • threshold: Distance threshold in meters for precision/recall (default: 0.05m)
  • correct_mesh_gravity: Whether to align ADT mesh gravity (default: False)
  • cut_height: Optional Z-height to slice meshes (default: None)
  • rnd_seed: Random seed for reproducibility (default: 42)

Returns:

  • metrics: Dictionary with 5 metrics (see below)
  • accuracy: Per-point accuracy array (N,) - distances from predicted points to GT
  • completeness: Per-point completeness array (M,) - distances from GT points to prediction

1.3.1.1 Understanding Accuracy vs. Completeness

The function computes two separate distance measurements by switching the roles of source and target:

Accuracy (Prediction → GT):

# Step 1: Sample points FROM predicted mesh
pred_pts, _ = trimesh.sample.sample_surface(pred_mesh, sample_num, seed=rnd_seed)

# Step 2: Compute distances TO ground truth mesh
accuracy = compute_pts_to_mesh_dist(pred_pts, gt_faces, gt_vertices, step)

Interpretation:

  • Measures how close predicted surface is to ground truth
  • High accuracy = low error = predicted points lie close to GT surface
  • Detects over-reconstruction (extra geometry in prediction)
  • Formula: \(\text{Accuracy} = \frac{1}{|\mathcal{P}|} \sum_{\mathbf{p} \in \mathcal{P}} \min_{\mathbf{q} \in \mathcal{M}_{\text{GT}}} \|\mathbf{p} - \mathbf{q}\|_2\)

Completeness (GT → Prediction):

# Step 1: Sample points FROM ground truth mesh
gt_pts, _ = trimesh.sample.sample_surface(gt_mesh, sample_num, seed=rnd_seed)

# Step 2: Compute distances TO predicted mesh
completeness = compute_pts_to_mesh_dist(gt_pts, pred_faces, pred_vertices, step)

Interpretation:

  • Measures how well prediction covers ground truth surface
  • High completeness = low error = GT points have nearby predicted points
  • Detects under-reconstruction (missing geometry in prediction)
  • Formula: \(\text{Completeness} = \frac{1}{|\mathcal{M}_{\text{GT}}|} \sum_{\mathbf{q} \in \mathcal{M}_{\text{GT}}} \min_{\mathbf{p} \in \mathcal{P}} \|\mathbf{p} - \mathbf{q}\|_2\)

Key Difference: Same algorithm (compute_pts_to_mesh_dist), but source and target are swapped.

1.3.1.2 Example Scenarios

Scenario Accuracy Completeness Interpretation
Perfect reconstruction Low ✅ Low ✅ Ideal case
Missing geometry (holes) Low ✅ High ❌ Under-reconstruction
Extra geometry (noise) High ❌ Low ✅ Over-reconstruction
Misaligned reconstruction High ❌ High ❌ Wrong pose/scale

1.3.1.3 Output Metrics Dictionary

metrics = {
    "Accuracy_mean_meters": np.mean(accuracy),        # Mean distance pred→GT
    "Completeness_mean_meters": np.mean(completeness), # Mean distance GT→pred
    "prec@0.05": precision_perc5,                      # % pred points < 5cm from GT
    "recal@0.05": recall_perc5,                        # % GT points < 5cm from pred
    "fscore@0.05": fscore_perc5,                       # Harmonic mean of prec & recall
}

Precision & Recall Computation:

# Precision: fraction of predicted points that are "correct" (within threshold)
precision_perc5 = np.mean((accuracy < 0.05).astype("float"))

# Recall: fraction of GT points that are "recovered" (within threshold)
recall_perc5 = np.mean((completeness < 0.05).astype("float"))

# F-score: harmonic mean balancing precision and recall
fscore_perc5 = 2 * precision_perc5 * recall_perc5 / (precision_perc5 + recall_perc5)

1.3.1.4 Why Both Directions Matter

Chamfer Distance combines both: \[ \text{CD}(\mathcal{P}, \mathcal{M}_{\text{GT}}) = \text{Accuracy} + \text{Completeness} \]

This bidirectional metric is crucial because: - Accuracy alone doesn’t penalize missing geometry - Completeness alone doesn’t penalize extra/noisy geometry - Together they provide complete picture of reconstruction quality

For RRI computation, we need CD to quantify improvement: \[ \text{RRI}(\mathbf{q}) = \frac{\text{CD}_{\text{before}} - \text{CD}_{\text{after}}}{\text{CD}_{\text{before}}} \]

1.3.2 compute_pts_to_mesh_dist()

Located in atek/evaluation/surface_reconstruction/surface_reconstruction_utils.py

Purpose: Compute minimum distance from points to mesh triangles.

Algorithm:

  1. Iterate through mesh faces in batches (for memory efficiency)
  2. For each point, compute:
    • Distance to triangle plane (if point projects inside triangle)
    • Distance to closest vertex (fallback)
  3. Use geometric projection test to determine if point projects onto triangle
  4. Return minimum distance across all triangles

Key sub-functions:

1.3.2.0.1 point_to_closest_tri_dist()

Computes point-to-triangle distances using barycentric coordinate projection:

Mathematical Details:

For each triangle defined by vertices \(\mathbf{v}_0, \mathbf{v}_1, \mathbf{v}_2\):

  1. Barycentric Coordinates: \[ \mathbf{u} = \mathbf{v}_1 - \mathbf{v}_0, \quad \mathbf{v} = \mathbf{v}_2 - \mathbf{v}_0, \quad \mathbf{w} = \mathbf{p} - \mathbf{v}_0 \]

    \[ \mathbf{n} = \mathbf{u} \times \mathbf{v}, \quad n^2 = \mathbf{n} \cdot \mathbf{n} \]

  2. Projection Test (check if point projects inside triangle): \[ \gamma = \frac{(\mathbf{u} \times \mathbf{w}) \cdot \mathbf{n}}{n^2}, \quad \beta = \frac{(\mathbf{w} \times \mathbf{v}) \cdot \mathbf{n}}{n^2}, \quad \alpha = 1 - \gamma - \beta \]

    Valid if: \(0 \leq \alpha, \beta, \gamma \leq 1\)

  3. Distance to Plane: \[ d = \left|\frac{\mathbf{n} \cdot \mathbf{w}}{\|\mathbf{n}\|}\right| \]

  4. Fallback: If point doesn’t project onto triangle, set distance to large value (1e6)

1.3.2.0.2 point_to_closest_vertex_dist()

Fallback method that computes Euclidean distance to nearest triangle vertex:

\[ d_{\text{vertex}}(\mathbf{p}) = \min_{i \in \{0,1,2\}} \|\mathbf{p} - \mathbf{v}_i\|_2 \]

1.4.2 Point Cloud Utilities (efm3d/utils/pointcloud.py)

1.4.2.1 Core Point Cloud Operations

1.4.2.1.1 get_points_world()

Converts depth images or point cloud data to world coordinates: - Handles ARIA depth maps via backprojection - Transforms from camera → rig → world coordinate frames - Filters invalid depth values (NaN, zero, or negative)

1.4.2.1.2 collapse_pointcloud_time()

Merges temporal point cloud sequences: - Flattens time dimension - Removes NaN values - Deduplicates points using torch.unique()

1.4.2.1.3 pointcloud_to_voxel_ids()

Converts 3D points to voxel grid indices:

Algorithm:

# Compute voxel size
dW = (x_max - x_min) / vW
dH = (y_max - y_min) / vH
dD = (z_max - z_min) / vD

# Convert point to voxel index
voxel_id = floor((point - voxel_min) / [dW, dH, dD])

Validation: Ensures points lie within voxel grid bounds

1.4.2.1.4 pointcloud_to_occupancy_snippet()

Creates occupancy grid from point cloud with free-space reasoning:

Three-phase process: 1. Camera origins: Mark as free space 2. Ray marching: Sample points along rays from camera to surface → mark as free 3. Surface points: Mark occupied at point locations

Parameters: - S: Number of samples along each ray (default: 1) - Uses conservative sampling: stops at depth - voxel_diagonal to avoid marking surface voxels as free

1.4.2.2 Occupancy Grid Construction

1.4.2.2.1 pointcloud_occupancy_samples()

Generates training samples for occupancy prediction:

Returns three point sets: - Occupied: Points beyond surface (at depth + voxel_diagonal) - Surface: Original point cloud points - Free space: Sampled points along rays before surface (at random depths < depth - voxel_diagonal)

Use case: Provides supervision for learning-based occupancy prediction

1.4.2.2.2 pointcloud_to_voxel_counts()

Converts point cloud to voxel grid with point density:

Output: Voxel grid where each cell contains count of points inside - Useful for understanding reconstruction density - Can be normalized and thresholded to create binary occupancy

1.4.3 Ray Utilities (efm3d/utils/ray.py)

1.4.3.1 Ray-Mesh Intersection

1.4.3.1.1 ray_obb_intersection()

Computes ray intersection with axis-aligned bounding boxes (voxel grids):

Algorithm (Slab method): 1. Test ray against 6 planes (±X, ±Y, ±Z) 2. For each plane: - Compute intersection parameter \(t\): \(t = \frac{(\mathbf{p}_{\text{plane}} - \mathbf{p}_{\text{ray}}) \cdot \mathbf{n}}{{\mathbf{d}_{\text{ray}}} \cdot \mathbf{n}}\) - Check if intersection point lies within plane bounds 3. Track minimum \(t_{\text{min}}\) (entry) and maximum \(t_{\text{max}}\) (exit) 4. Valid intersection if \(t_{\text{min}} < t_{\text{max}}\)

Returns: \((t_{\text{min}}, t_{\text{max}})\) or optionally 3D intersection points

1.4.3.1.2 sample_depths_in_grid()

Samples depths along rays within voxel grid bounds:

Use cases: - Free-space sampling for occupancy supervision - NeRF-style volumetric rendering - Visibility reasoning

Modes: - random: Add noise to uniformly-spaced samples - uniform: Evenly-spaced samples along ray

1.4.4 Reconstruction Utilities (efm3d/utils/reconstruction.py)

1.4.4.1 compute_occupancy_loss_subvoxel()

Training loss for occupancy prediction:

Strategy:

  1. Sample occupied, surface, and free-space points
  2. Query predicted occupancy via trilinear interpolation
  3. Compute loss against ground truth labels

Loss types:

  • focal: Focal loss (handles class imbalance)
  • ce: Binary cross-entropy
  • l1, l2: Regression losses
  • logl1: Log-space L1

Key feature: Subsampling support to reduce memory usage

1.4.5 Depth Utilities (efm3d/utils/depth.py)

1.4.5.1 dist_im_to_point_cloud_im()

Converts depth images to 3D point clouds:

Algorithm:

  1. Generate ray grid for camera using ray_grid()
  2. Project rays to depth: \(\mathbf{p}_{\text{3D}} = \mathbf{o}_{\text{ray}} + d \cdot \mathbf{d}_{\text{ray}}\)
  3. Transform from rig coordinates to camera coordinates
  4. Filter invalid depths (≤ 0)

Supports: Batched operation over time and batch dimensions

1.4.6 Voxel Sampling Utilities (efm3d/utils/voxel_sampling.py)

1.4.6.1 sample_voxels()

Samples voxel grid features at 3D point locations using trilinear interpolation:

Implementation:

  • Normalizes points to \([-1, 1]\) range
  • Uses torch.nn.functional.grid_sample() (non-differentiable w.r.t. points)
  • Alternative: diff_grid_sample() for differentiable sampling

Use case: Query occupancy or features at arbitrary 3D locations

1.4.6.2 pc_to_vox()

Converts point cloud coordinates to normalized voxel coordinates:

Handles:

  • List-based or tensor-based voxel extents
  • Batched point clouds
  • Validity checking (points within bounds)

1.5 Computing RRI with Ground Truth Meshes

1.5.1 RRI Definition

For Next-Best-View planning, the Relative Reconstruction Improvement (RRI) quantifies the expected reconstruction quality gain from capturing a new view:

\[ \text{RRI}(\mathbf{q}) = \frac{\text{CD}(\mathcal{P}_t, \mathcal{M}_{\text{GT}}) - \text{CD}(\mathcal{P}_{t \cup \mathbf{q}}, \mathcal{M}_{\text{GT}})}{\text{CD}(\mathcal{P}_t, \mathcal{M}_{\text{GT}})} \]

Where:

  • \(\mathbf{q}\): Candidate viewpoint (5DoF: position + yaw, pitch)
  • \(\mathcal{P}_t\): Current reconstruction from first \(t\) views
  • \(\mathcal{P}_{t \cup \mathbf{q}}\): Updated reconstruction after capturing from \(\mathbf{q}\)
  • \(\mathcal{M}_{\text{GT}}\): Ground truth mesh (from ASE .ply files)

Properties:

  • Range: \([0, 1]\) where higher = more improvement
  • Normalized by current error → scale-independent
  • Requires GT only during training

1.5.2 RRI Oracle Pipeline for ASE Dataset

Step 1: Load Ground Truth Mesh

gt_mesh = trimesh.load("scene_id/gt_mesh.ply")

Step 2: Construct Current Reconstruction \(\mathcal{P}_t\)

Option A: Dense point cloud from depth maps

# For each captured view i in [1, ..., t]
for depth_map, camera, pose in views:
    pc_camera = dist_im_to_point_cloud_im(depth_map, camera)
    pc_world = pose.transform(pc_camera)
    P_t.append(pc_world)
P_t = merge_and_filter(P_t)  # Remove duplicates, NaNs

Option B: Semi-dense SLAM point cloud

# Use ASE's semidense_points.csv.gz
P_t = load_semidense_points("scene_id/semidense_points.csv.gz")

Step 3: Simulate Observation from Candidate View \(\mathbf{q}\)

Option 3A: Ray-casting to GT mesh

# Generate ray grid for candidate camera
rays_camera = ray_grid(candidate_camera)
rays_world = q_pose.transform_rays(rays_camera)

# Intersect with GT mesh
intersection_points = ray_mesh_intersection(rays_world, gt_mesh)
P_q = intersection_points  # New points visible from q

Option 3B: Render depth from GT mesh

# Use mesh rendering to generate depth map
depth_q = render_depth(gt_mesh, q_pose, candidate_camera)
pc_q = dist_im_to_point_cloud_im(depth_q, candidate_camera)
P_q = q_pose.transform(pc_q)

Step 4: Merge Reconstructions

P_t_union_q = merge_pointclouds(P_t, P_q)
P_t_union_q = voxel_downsample(P_t_union_q, voxel_size=0.01)  # Consistency

Step 5: Compute Chamfer Distances

# Sample points from GT mesh
M_GT = trimesh.sample.sample_surface(gt_mesh, count=10000)

# Compute CD before new view
cd_before = chamfer_distance(P_t, M_GT)

# Compute CD after new view
cd_after = chamfer_distance(P_t_union_q, M_GT)

# Compute RRI
rri = (cd_before - cd_after) / cd_before

Note on Chamfer Distance: Can use ATEK’s compute_pts_to_mesh_dist() or EFM3D’s eval_mesh_to_mesh().

1.5.3 Important Considerations

1.5.3.1 Point Cloud Density

  • Issue: CD depends on point cloud density
  • Solution: Use consistent voxel downsampling (e.g., 1cm voxels)
  • Alternative: Normalize point counts or use mesh-to-mesh CD after reconstruction

1.5.3.2 Visibility Information

  • ASE provides semidense_observations.csv.gz mapping points to observing frames
  • Can be used to identify which points are newly observed from \(\mathbf{q}\)
  • Useful for more accurate simulation without full ray-casting

1.5.3.3 Entity-Wise RRI

For scene-level reconstruction with semantic understanding:

Decompose by object category:

for entity_class in ["chair", "table", "wall", ...]:
    # Segment GT mesh by semantic class
    M_GT_entity = segment_mesh(gt_mesh, entity_class)

    # Filter reconstruction points by entity
    P_t_entity = filter_by_entity(P_t, entity_class)
    P_t_union_q_entity = filter_by_entity(P_t_union_q, entity_class)

    # Compute per-entity RRI
    rri_entity = compute_rri(P_t_entity, P_t_union_q_entity, M_GT_entity)

Benefits: - Prioritize under-reconstructed object categories - Combine with semantic embeddings for multi-task learning - Align with task-driven exploration objectives

# Reconstruction Quality Metrics & RRI Computation

This document provides a comprehensive overview of reconstruction quality metrics and how to compute the Relative Reconstruction Improvement (RRI) score for Next-Best-View planning.

## Surface Reconstruction Metrics

### Overview

The evaluation of 3D reconstruction quality relies on comparing predicted reconstructions against ground truth meshes. Both **ATEK** and **EFM3D** provide implementations for computing these metrics using:

- **[Trimesh](https://github.com/mikedh/trimesh)**: For mesh loading and surface sampling
- **PyTorch**: For efficient distance computations
- **Point-to-mesh distance functions**: Core geometric primitives

### Key Metrics

#### 1. Accuracy (Prediction → Ground Truth)

Measures how close predicted points are to the ground truth surface.

$$
\text{Accuracy} = \frac{1}{|\mathcal{P}|} \sum_{\mathbf{p} \in \mathcal{P}} \min_{\mathbf{q} \in \mathcal{M}_{\text{GT}}} \|\mathbf{p} - \mathbf{q}\|_2
$$

- $\mathcal{P}$: Sampled points from predicted mesh/reconstruction
- $\mathcal{M}_{\text{GT}}$: Sampled points from ground truth mesh
- Lower values indicate better reconstruction accuracy

#### 2. Completeness (Ground Truth → Prediction)

Measures how well the prediction covers the ground truth surface.

$$
\text{Completeness} = \frac{1}{|\mathcal{M}_{\text{GT}}|} \sum_{\mathbf{q} \in \mathcal{M}_{\text{GT}}} \min_{\mathbf{p} \in \mathcal{P}} \|\mathbf{p} - \mathbf{q}\|_2
$$

- Lower values indicate better coverage of the ground truth

#### 3. Chamfer Distance (Bidirectional)

Combines both accuracy and completeness:

$$
\text{CD}(\mathcal{P}, \mathcal{M}_{\text{GT}}) = \text{Accuracy} + \text{Completeness}
$$

This symmetric metric captures both over-reconstruction (accuracy) and under-reconstruction (completeness).

#### 4. Precision, Recall & F-score

Binary classification metrics at threshold $\tau$ (typically 5cm):

$$
\text{Precision}_\tau = \frac{|\{\mathbf{p} \in \mathcal{P} : \min_{\mathbf{q} \in \mathcal{M}_{\text{GT}}} \|\mathbf{p} - \mathbf{q}\| < \tau\}|}{|\mathcal{P}|}
$$

$$
\text{Recall}_\tau = \frac{|\{\mathbf{q} \in \mathcal{M}_{\text{GT}}: \min_{\mathbf{p} \in \mathcal{P}} \|\mathbf{p} - \mathbf{q}\| < \tau\}|}{|\mathcal{M}_{\text{GT}}|}
$$

$$
\text{F-score}_\tau = \frac{2 \cdot \text{Precision} \cdot \text{Recall}}{\text{Precision} + \text{Recall}}
$$

## ATEK Implementation

### `evaluate_single_mesh_pair()`

Located in [`atek/evaluation/surface_reconstruction/surface_reconstruction_metrics.py`](https://github.com/facebookresearch/ATEK/blob/main/atek/evaluation/surface_reconstruction/surface_reconstruction_metrics.py).

**Purpose**: Complete evaluation pipeline that computes all reconstruction quality metrics between a predicted mesh and ground truth mesh.

**Workflow**:

1. **Load meshes**: Uses `trimesh.load()` to load both predicted and GT meshes
2. **(Optional) Gravity correction**: For ADT datasets, aligns gravity direction via `correct_adt_mesh_gravity()`
3. **(Optional) Height slicing**: Cuts meshes at specified height using plane slicing
4. **Sample surfaces**: Uniformly samples `sample_num` points from both meshes using `trimesh.sample.sample_surface()`
   - Seeds RNG for reproducibility
   - Poisson disk sampling ensures uniform distribution
5. **Compute bidirectional distances**: Calls `compute_pts_to_mesh_dist()` twice (see below for details)
6. **Calculate metrics**: Computes accuracy, completeness, precision@5cm, recall@5cm, F-score@5cm

**Parameters**:

- `pred_mesh_filename`: Path to predicted mesh (`.ply`, `.obj`, etc.)
- `gt_mesh_filename`: Path to ground truth mesh
- `sample_num`: Number of points to sample (default: 10,000)
- `step`: Batch size for distance computation (default: 50,000 triangles)
- `threshold`: Distance threshold in meters for precision/recall (default: 0.05m)
- `correct_mesh_gravity`: Whether to align ADT mesh gravity (default: False)
- `cut_height`: Optional Z-height to slice meshes (default: None)
- `rnd_seed`: Random seed for reproducibility (default: 42)

**Returns**:

- `metrics`: Dictionary with 5 metrics (see below)
- `accuracy`: Per-point accuracy array (N,) - distances from predicted points to GT
- `completeness`: Per-point completeness array (M,) - distances from GT points to prediction

#### Understanding Accuracy vs. Completeness

The function computes **two separate distance measurements** by switching the roles of source and target:

**Accuracy (Prediction → GT):**
```python
# Step 1: Sample points FROM predicted mesh
pred_pts, _ = trimesh.sample.sample_surface(pred_mesh, sample_num, seed=rnd_seed)

# Step 2: Compute distances TO ground truth mesh
accuracy = compute_pts_to_mesh_dist(pred_pts, gt_faces, gt_vertices, step)
```

**Interpretation**:

- Measures how close predicted surface is to ground truth
- High accuracy = low error = predicted points lie close to GT surface
- Detects **over-reconstruction** (extra geometry in prediction)
- Formula: $\text{Accuracy} = \frac{1}{|\mathcal{P}|} \sum_{\mathbf{p} \in \mathcal{P}} \min_{\mathbf{q} \in \mathcal{M}_{\text{GT}}} \|\mathbf{p} - \mathbf{q}\|_2$

**Completeness (GT → Prediction):**
```python
# Step 1: Sample points FROM ground truth mesh
gt_pts, _ = trimesh.sample.sample_surface(gt_mesh, sample_num, seed=rnd_seed)

# Step 2: Compute distances TO predicted mesh
completeness = compute_pts_to_mesh_dist(gt_pts, pred_faces, pred_vertices, step)
```

**Interpretation**:

- Measures how well prediction covers ground truth surface
- High completeness = low error = GT points have nearby predicted points
- Detects **under-reconstruction** (missing geometry in prediction)
- Formula: $\text{Completeness} = \frac{1}{|\mathcal{M}_{\text{GT}}|} \sum_{\mathbf{q} \in \mathcal{M}_{\text{GT}}} \min_{\mathbf{p} \in \mathcal{P}} \|\mathbf{p} - \mathbf{q}\|_2$

**Key Difference**: Same algorithm (`compute_pts_to_mesh_dist`), **but source and target are swapped**.

#### Example Scenarios

| Scenario | Accuracy | Completeness | Interpretation |
|----------|----------|--------------|----------------|
| Perfect reconstruction | Low ✅ | Low ✅ | Ideal case |
| Missing geometry (holes) | Low ✅ | High ❌ | Under-reconstruction |
| Extra geometry (noise) | High ❌ | Low ✅ | Over-reconstruction |
| Misaligned reconstruction | High ❌ | High ❌ | Wrong pose/scale |

#### Output Metrics Dictionary

```python
metrics = {
    "Accuracy_mean_meters": np.mean(accuracy),        # Mean distance pred→GT
    "Completeness_mean_meters": np.mean(completeness), # Mean distance GT→pred
    "prec@0.05": precision_perc5,                      # % pred points < 5cm from GT
    "recal@0.05": recall_perc5,                        # % GT points < 5cm from pred
    "fscore@0.05": fscore_perc5,                       # Harmonic mean of prec & recall
}
```

**Precision & Recall Computation**:
```python
# Precision: fraction of predicted points that are "correct" (within threshold)
precision_perc5 = np.mean((accuracy < 0.05).astype("float"))

# Recall: fraction of GT points that are "recovered" (within threshold)
recall_perc5 = np.mean((completeness < 0.05).astype("float"))

# F-score: harmonic mean balancing precision and recall
fscore_perc5 = 2 * precision_perc5 * recall_perc5 / (precision_perc5 + recall_perc5)
```

#### Why Both Directions Matter

**Chamfer Distance** combines both:
$$
\text{CD}(\mathcal{P}, \mathcal{M}_{\text{GT}}) = \text{Accuracy} + \text{Completeness}
$$

This bidirectional metric is crucial because:
- **Accuracy alone** doesn't penalize missing geometry
- **Completeness alone** doesn't penalize extra/noisy geometry
- **Together** they provide complete picture of reconstruction quality

**For RRI computation**, we need CD to quantify improvement:
$$
\text{RRI}(\mathbf{q}) = \frac{\text{CD}_{\text{before}} - \text{CD}_{\text{after}}}{\text{CD}_{\text{before}}}
$$

### `compute_pts_to_mesh_dist()`

Located in `atek/evaluation/surface_reconstruction/surface_reconstruction_utils.py`

**Purpose**: Compute minimum distance from points to mesh triangles.

**Algorithm**:

1. Iterate through mesh faces in batches (for memory efficiency)
2. For each point, compute:
   - Distance to triangle plane (if point projects inside triangle)
   - Distance to closest vertex (fallback)
3. Use geometric projection test to determine if point projects onto triangle
4. Return minimum distance across all triangles

**Key sub-functions**:

##### `point_to_closest_tri_dist()`

Computes point-to-triangle distances using barycentric coordinate projection:

**Mathematical Details**:

For each triangle defined by vertices $\mathbf{v}_0, \mathbf{v}_1, \mathbf{v}_2$:

1. **Barycentric Coordinates**:
   $$
   \mathbf{u} = \mathbf{v}_1 - \mathbf{v}_0, \quad \mathbf{v} = \mathbf{v}_2 - \mathbf{v}_0, \quad \mathbf{w} = \mathbf{p} - \mathbf{v}_0
   $$

   $$
   \mathbf{n} = \mathbf{u} \times \mathbf{v}, \quad n^2 = \mathbf{n} \cdot \mathbf{n}
   $$

2. **Projection Test** (check if point projects inside triangle):
   $$
   \gamma = \frac{(\mathbf{u} \times \mathbf{w}) \cdot \mathbf{n}}{n^2}, \quad
   \beta = \frac{(\mathbf{w} \times \mathbf{v}) \cdot \mathbf{n}}{n^2}, \quad
   \alpha = 1 - \gamma - \beta
   $$

   Valid if: $0 \leq \alpha, \beta, \gamma \leq 1$

3. **Distance to Plane**:
   $$
   d = \left|\frac{\mathbf{n} \cdot \mathbf{w}}{\|\mathbf{n}\|}\right|
   $$

4. **Fallback**: If point doesn't project onto triangle, set distance to large value (1e6)

##### `point_to_closest_vertex_dist()`

Fallback method that computes Euclidean distance to nearest triangle vertex:

$$
d_{\text{vertex}}(\mathbf{p}) = \min_{i \in \{0,1,2\}} \|\mathbf{p} - \mathbf{v}_i\|_2
$$

**Purpose**: Comprehensive mesh-to-mesh evaluation with visualizations.

**Enhancements over ATEK**:

- `viz`: Dictionary with colored point clouds for visualization
- `raw_data`: Per-point accuracy and completeness arrays

### Point Cloud Utilities (`efm3d/utils/pointcloud.py`)

#### Core Point Cloud Operations

##### `get_points_world()`

Converts depth images or point cloud data to world coordinates:
- Handles ARIA depth maps via backprojection
- Transforms from camera → rig → world coordinate frames
- Filters invalid depth values (NaN, zero, or negative)

##### `collapse_pointcloud_time()`

Merges temporal point cloud sequences:
- Flattens time dimension
- Removes NaN values
- Deduplicates points using `torch.unique()`

##### `pointcloud_to_voxel_ids()`

Converts 3D points to voxel grid indices:

**Algorithm**:
```python
# Compute voxel size
dW = (x_max - x_min) / vW
dH = (y_max - y_min) / vH
dD = (z_max - z_min) / vD

# Convert point to voxel index
voxel_id = floor((point - voxel_min) / [dW, dH, dD])
```

**Validation**: Ensures points lie within voxel grid bounds

##### `pointcloud_to_occupancy_snippet()`

Creates occupancy grid from point cloud with free-space reasoning:

**Three-phase process**:
1. **Camera origins**: Mark as free space
2. **Ray marching**: Sample points along rays from camera to surface → mark as free
3. **Surface points**: Mark occupied at point locations

**Parameters**:
- `S`: Number of samples along each ray (default: 1)
- Uses conservative sampling: stops at `depth - voxel_diagonal` to avoid marking surface voxels as free

#### Occupancy Grid Construction

##### `pointcloud_occupancy_samples()`

Generates training samples for occupancy prediction:

**Returns three point sets**:
- **Occupied**: Points beyond surface (at `depth + voxel_diagonal`)
- **Surface**: Original point cloud points
- **Free space**: Sampled points along rays before surface (at random depths < `depth - voxel_diagonal`)

**Use case**: Provides supervision for learning-based occupancy prediction

##### `pointcloud_to_voxel_counts()`

Converts point cloud to voxel grid with point density:

**Output**: Voxel grid where each cell contains count of points inside
- Useful for understanding reconstruction density
- Can be normalized and thresholded to create binary occupancy

### Ray Utilities (`efm3d/utils/ray.py`)

#### Ray-Mesh Intersection

##### `ray_obb_intersection()`

Computes ray intersection with axis-aligned bounding boxes (voxel grids):

**Algorithm** (Slab method):
1. Test ray against 6 planes (±X, ±Y, ±Z)
2. For each plane:
   - Compute intersection parameter $t$: $t = \frac{(\mathbf{p}_{\text{plane}} - \mathbf{p}_{\text{ray}}) \cdot \mathbf{n}}{{\mathbf{d}_{\text{ray}}} \cdot \mathbf{n}}$
   - Check if intersection point lies within plane bounds
3. Track minimum $t_{\text{min}}$ (entry) and maximum $t_{\text{max}}$ (exit)
4. Valid intersection if $t_{\text{min}} < t_{\text{max}}$

**Returns**: $(t_{\text{min}}, t_{\text{max}})$ or optionally 3D intersection points

##### `sample_depths_in_grid()`

Samples depths along rays within voxel grid bounds:

**Use cases**:
- Free-space sampling for occupancy supervision
- NeRF-style volumetric rendering
- Visibility reasoning

**Modes**:
- `random`: Add noise to uniformly-spaced samples
- `uniform`: Evenly-spaced samples along ray

### Reconstruction Utilities (`efm3d/utils/reconstruction.py`)

#### `compute_occupancy_loss_subvoxel()`

Training loss for occupancy prediction:

**Strategy**:

1. Sample occupied, surface, and free-space points
2. Query predicted occupancy via trilinear interpolation
3. Compute loss against ground truth labels

**Loss types**:

- `focal`: Focal loss (handles class imbalance)
- `ce`: Binary cross-entropy
- `l1`, `l2`: Regression losses
- `logl1`: Log-space L1

**Key feature**: Subsampling support to reduce memory usage

### Depth Utilities (`efm3d/utils/depth.py`)

#### `dist_im_to_point_cloud_im()`

Converts depth images to 3D point clouds:

**Algorithm**:

1. Generate ray grid for camera using `ray_grid()`
2. Project rays to depth: $\mathbf{p}_{\text{3D}} = \mathbf{o}_{\text{ray}} + d \cdot \mathbf{d}_{\text{ray}}$
3. Transform from rig coordinates to camera coordinates
4. Filter invalid depths (≤ 0)

**Supports**: Batched operation over time and batch dimensions

### Voxel Sampling Utilities (`efm3d/utils/voxel_sampling.py`)

#### `sample_voxels()`

Samples voxel grid features at 3D point locations using trilinear interpolation:

**Implementation**:

- Normalizes points to $[-1, 1]$ range
- Uses `torch.nn.functional.grid_sample()` (non-differentiable w.r.t. points)
- Alternative: `diff_grid_sample()` for differentiable sampling

**Use case**: Query occupancy or features at arbitrary 3D locations

#### `pc_to_vox()`

Converts point cloud coordinates to normalized voxel coordinates:

**Handles**:

- List-based or tensor-based voxel extents
- Batched point clouds
- Validity checking (points within bounds)

## Computing RRI with Ground Truth Meshes

### RRI Definition

For Next-Best-View planning, the **Relative Reconstruction Improvement (RRI)** quantifies the expected reconstruction quality gain from capturing a new view:

$$
\text{RRI}(\mathbf{q}) = \frac{\text{CD}(\mathcal{P}_t, \mathcal{M}_{\text{GT}}) - \text{CD}(\mathcal{P}_{t \cup \mathbf{q}}, \mathcal{M}_{\text{GT}})}{\text{CD}(\mathcal{P}_t, \mathcal{M}_{\text{GT}})}
$$

**Where**:

- $\mathbf{q}$: Candidate viewpoint (5DoF: position + yaw, pitch)
- $\mathcal{P}_t$: Current reconstruction from first $t$ views
- $\mathcal{P}_{t \cup \mathbf{q}}$: Updated reconstruction after capturing from $\mathbf{q}$
- $\mathcal{M}_{\text{GT}}$: Ground truth mesh (from ASE `.ply` files)

**Properties**:

- Range: $[0, 1]$ where higher = more improvement
- Normalized by current error → scale-independent
- Requires GT only during training

### RRI Oracle Pipeline for ASE Dataset

**Step 1: Load Ground Truth Mesh**
```python
gt_mesh = trimesh.load("scene_id/gt_mesh.ply")
```

**Step 2: Construct Current Reconstruction $\mathcal{P}_t$**

**Option A: Dense point cloud from depth maps**
```python
# For each captured view i in [1, ..., t]
for depth_map, camera, pose in views:
    pc_camera = dist_im_to_point_cloud_im(depth_map, camera)
    pc_world = pose.transform(pc_camera)
    P_t.append(pc_world)
P_t = merge_and_filter(P_t)  # Remove duplicates, NaNs
```

**Option B: Semi-dense SLAM point cloud**
```python
# Use ASE's semidense_points.csv.gz
P_t = load_semidense_points("scene_id/semidense_points.csv.gz")
```

**Step 3: Simulate Observation from Candidate View $\mathbf{q}$**

**Option 3A: Ray-casting to GT mesh**
```python
# Generate ray grid for candidate camera
rays_camera = ray_grid(candidate_camera)
rays_world = q_pose.transform_rays(rays_camera)

# Intersect with GT mesh
intersection_points = ray_mesh_intersection(rays_world, gt_mesh)
P_q = intersection_points  # New points visible from q
```

**Option 3B: Render depth from GT mesh**
```python
# Use mesh rendering to generate depth map
depth_q = render_depth(gt_mesh, q_pose, candidate_camera)
pc_q = dist_im_to_point_cloud_im(depth_q, candidate_camera)
P_q = q_pose.transform(pc_q)
```

**Step 4: Merge Reconstructions**
```python
P_t_union_q = merge_pointclouds(P_t, P_q)
P_t_union_q = voxel_downsample(P_t_union_q, voxel_size=0.01)  # Consistency
```

**Step 5: Compute Chamfer Distances**
```python
# Sample points from GT mesh
M_GT = trimesh.sample.sample_surface(gt_mesh, count=10000)

# Compute CD before new view
cd_before = chamfer_distance(P_t, M_GT)

# Compute CD after new view
cd_after = chamfer_distance(P_t_union_q, M_GT)

# Compute RRI
rri = (cd_before - cd_after) / cd_before
```

**Note on Chamfer Distance**: Can use ATEK's `compute_pts_to_mesh_dist()` or EFM3D's `eval_mesh_to_mesh()`.

### Important Considerations

#### Point Cloud Density
- **Issue**: CD depends on point cloud density
- **Solution**: Use consistent voxel downsampling (e.g., 1cm voxels)
- **Alternative**: Normalize point counts or use mesh-to-mesh CD after reconstruction

#### Visibility Information
- ASE provides `semidense_observations.csv.gz` mapping points to observing frames
- Can be used to identify which points are newly observed from $\mathbf{q}$
- Useful for more accurate simulation without full ray-casting

#### Entity-Wise RRI
For scene-level reconstruction with semantic understanding:

**Decompose by object category**:
```python
for entity_class in ["chair", "table", "wall", ...]:
    # Segment GT mesh by semantic class
    M_GT_entity = segment_mesh(gt_mesh, entity_class)

    # Filter reconstruction points by entity
    P_t_entity = filter_by_entity(P_t, entity_class)
    P_t_union_q_entity = filter_by_entity(P_t_union_q, entity_class)

    # Compute per-entity RRI
    rri_entity = compute_rri(P_t_entity, P_t_union_q_entity, M_GT_entity)
```

**Benefits**:
- Prioritize under-reconstructed object categories
- Combine with semantic embeddings for multi-task learning
- Align with task-driven exploration objectives
